Technical design

How the four pillars actually get built on the Level Up stack as it exists today: Flutter + Firebase, a service-locator architecture, a live chat pipeline, and a structured workout library. This is the engineering companion to the vision - concrete enough to scope and estimate from, honest about the open decisions.

Design stance

One new server-side AI service layer (Cloud Functions) + one new client-side AiService in level_up_shared, registered in the locator. Every pillar is a feature on top of those two additions. We add a brain to the existing body.

The shape of it

Architecture at a glance

Keys and model calls live behind Cloud Functions. The app never holds an API key and never calls a model directly.

CLIENTFlutter appAiService (locator) → callable function. Streams results into StreamBuilder UI.
→
FUNCTIONSAI service layerAuth check, rate-limit, retrieve context, build prompt, call model, moderate, persist.
→
RETRIEVALKnowledge + libraryVector store of coach knowledge + Firestore workout/exercise data.
→
MODELLLM / visionText model for chat/agents; on-device pose for video.

Why server-side

API keys never ship in the app bundle (trivially extractable). Rate-limiting, cost control, prompt assembly, moderation, and audit logging all live in one place the client can't bypass. The existing functions setup (TypeScript, deployed to us-central1) is where this goes.

Why a shared AiService

All three apps need AI surfaces. Putting AiService in level_up_shared beside ChatService/WorkoutsService and registering it in the locator means one implementation, three consumers - consistent with how every other service already works.

Open decision #1

Which model(s)

Two independent choices: the text model (pillars 01-03) and the vision/pose stack (pillar 04). They don't have to come from the same vendor.

Text model (chat companion, coach assistant, workout generation)

Google Gemini (Vertex AI / Firebase AI Logic)Strongest defaultNative Firebase integration (Firebase AI Logic SDK), same GCP project & billing as level-up-e4147, App Check support, strong tool-use. Lowest integration friction given we're already all-in on Firebase.

Recommendation

Start with Gemini via Firebase AI Logic / Vertex for lowest friction and unified billing, but isolate the model behind a thin LlmProvider interface in the functions layer so swapping to Claude/GPT for a given task is a config change, not a rewrite. Model quality moves fast; don't hard-couple to one vendor.

Vision / pose stack - covered in pillar 04

On-device pose estimation (Google ML Kit Pose Detection or MediaPipe) rather than a cloud vision API, for both cost and privacy. Full rationale on the form-coaching page.

Pillar 01 - the foundation

Coach knowledge & retrieval (RAG)

A retrieval-augmented pipeline so answers are grounded in Benson's actual coaching, not generic fitness advice. Everything in pillars 02 and 03 grounds against this same store.

The pipeline

1 · INGESTCaptureVoice memos (transcribed), brain-dump prompts, existing exercise descriptions + coachNotes, past chat messages, imported docs.
→
2 · CHUNK + EMBEDIndexSplit into passages, embed to vectors, store with metadata (topic, drill, skill level).
→
3 · RETRIEVESearchOn a question, embed the query, pull top-k relevant passages.
→
4 · GENERATEGroundPass retrieved passages + question to the LLM; answer cites/uses only that context.

Where the vectors live

  • Firestore vector search (native find_nearest / vector index) - keeps everything in the existing database and security model. Lowest new-infrastructure cost. Recommended starting point for a coach-sized corpus.
  • Vertex AI Vector Search - scales further, more retrieval features, more setup. Reach for it only if the corpus or query volume outgrows Firestore vectors.
  • Managed RAG (Vertex RAG Engine) - offloads chunking/embedding/retrieval if we'd rather not own the pipeline.
The real problem is capture, not retrieval

RAG is well-trodden. The feature succeeds or fails on whether knowledge actually gets in with near-zero effort from Benson. Prioritize the low-friction capture surfaces (voice → transcript, one-tap "explain this drill" prompts, auto-mining existing coachNote and chat history) over retrieval sophistication. A great retriever over an empty store answers nothing.

Proposed Firestore shape 
coachKnowledge/{docId}
  coachId: string
  source: 'voice' | 'chat' | 'exercise' | 'manual' | 'import'
  topic: string          // e.g. "shooting", "defense"
  text: string           // the passage
  embedding: vector<768> // for find_nearest
  createdAt, updatedAt
  status: 'active' | 'archived'   // curate-able

Embedding generated in a Function on write (Firestore trigger), mirroring the existing resize-images on-upload pattern. Per-coach scoping in security rules.

Pillars 02 & 03 - chat & agents

Conversational + agentic layer

Riding the existing chat pipeline

The companion is a new author in the existing conversations/{id}/messages stream. Minimal model change: extend the sealed ChatMessage hierarchy (or add an authorType) so the UI can render an AI-authored message distinctly and so we never confuse an AI draft with a human send.

Message model delta 
// existing: sealed ChatMessage -> TextChatMessage | VideoChatMessage
// add a provenance marker rather than a whole new collection:
authorType: 'human' | 'aiCompanion'   // on the message
status?:    'draft' | 'sent'          // coach-side drafts await approval

A coach-side draft is a message with status: 'draft' that is never delivered until Benson taps approve - this is the human-in-the-loop gate for pillar 02's draft-reply, shared with pillar 03's action approvals.

Agentic actions (pillar 03) = tool use behind an approval gate

The agent is given a small, explicit allow-list of tools (functions) it may call - e.g. draftWorkout, proposeAssignment, summariseAthlete. The model decides which tool and with what arguments; the Function executes only allow-listed tools, and anything athlete-facing is staged as a proposal that Benson approves.

ToolRiskGate
summariseAthlete, suggestDrillsRead-onlyNone - safe to run automatically
draftWorkout, draftReplyCreates a draftVisible to coach; nothing sent/assigned
assignWorkout, sendMessageAthlete-facing writeExplicit per-action approval + audit log entry
Audit + guardrails

Every agent action writes an aiActions/{id} audit record (who, what tool, args, approved-by, when). Athlete-facing writes are impossible without an approval token. This is enforced in the Function, not the client - the gate is real only when the server rejects an unapproved write.

Pillar 04 - the flagship

Pose & action detection

On-device first. Raw video stays on the athlete's phone; only abstract skeleton data or feedback leaves, and only if needed.

Pose estimation - on device

Google ML Kit Pose Detection or MediaPipe Pose, running in the Flutter app via platform plugins. Produces per-frame skeletons (33 landmarks) locally. No raw video upload required. Free at the point of use - no per-call cloud vision cost.

Action / form analysis

Interpret the skeleton over time: joint angles, tempo, range of motion, symmetry, rep counting. Start with a small set of rule-based form checks per movement (squat depth, knee valgus on landing, shooting-elbow alignment) before any learned action-recognition model. Ships value with far less ML risk.

Why on-device matters here (cost AND privacy)

  • Cost: cloud video inference is expensive per minute; on-device pose is effectively free and scales to every rep without a per-rep bill.
  • Privacy: raw video of (often young) athletes never leaves the phone. Only the derived skeleton/feedback is optionally synced - and only with consent. This is the trust differentiator, detailed on the form-coaching page.
  • Latency: instant on-device feedback during/after the rep, no round-trip.

Cross-cutting concerns

The things every pillar needs

💸 Cost & rate limiting

Per-user and per-coach quotas enforced in the Function. Cache embeddings and common answers. Prefer cheaper/smaller models for routine tasks, escalate to larger models only for hard ones. On-device for vision keeps the biggest cost center off the bill entirely.

🛡 Safety & moderation

Input + output moderation on the companion. Hard rules: no medical/injury/diagnosis advice - defer to coach or a professional. Grounding + an explicit "I'm not sure, ask your coach" fallback to curb hallucination. Extra care for minors.

🔐 Security & access

Firestore rules scope knowledge per coach and conversations per participant. App Check on the AI functions to block off-app abuse. Agent writes gated by approval tokens server-side. Keys only in Functions config / Secret Manager.

📊 Evaluation

Before shipping the companion: a small eval set of real athlete questions + Benson's gold answers, scored for groundedness and tone. For pose: hand-labelled clips per movement to measure form-flag accuracy. Don't claim it works until measured against real cases.

Build-order rationale

The roadmap sequences these as Capture → Converse → Act → See for a reason: each leans on the last. The knowledge base (01) grounds the companion (02); the companion's draft-approve gate (02) is the same gate the agent needs (03); and the vision pillar (04) is best last because feedback is most useful when it can reference Benson's standards, and because it carries the highest privacy and ML-accuracy bar.

Honest unknowns

Open decisions to make before Wave 1

  • Model vendor for text - Gemini (recommended for Firebase-native fit) vs Claude/GPT. Behind a provider interface either way.
  • Vector store - Firestore native vectors (recommended start) vs Vertex Vector Search vs managed RAG.
  • Capture surfaces - which one or two we build first (voice-to-text is highest leverage, but needs a transcription step).
  • Persona - how much of Benson's voice/tone we configure explicitly vs let emerge from the knowledge base.
  • Cost ceiling - a monthly AI budget that sets the quota math.
  • Content seeding - a focused session with Benson to seed the initial knowledge base and triage the existing thin/placeholder Firestore content.
  • Consent & minors - the consent + parental-consent flows for pillar 04 should be specced before any video feature, not after.
